## Project Exploration Report: shortless-ios

### 1. Executive Summary

The `shortless-ios` project is a modern, multi-target iOS application built with Swift. Its architecture is centered around a main application that serves as a container and configuration UI for numerous app extensions. Each extension provides a distinct piece of functionality, such as Safari content blocking, Screen Time integration, and widgets.

Code sharing and decoupling are achieved through a local Swift Package, `ShortlessKit`, which centralizes shared models, constants, and business logic. Data is shared between the main app and its extensions via a shared App Group and `UserDefaults`, a standard and effective pattern for inter-process communication on iOS. The project structure itself is managed by XcodeGen via a `project.yml` file, ensuring a consistent and maintainable build configuration.

### 2. Build System & Project Configuration

The project's structure and Xcode workspace are defined in `project.yml` and generated by **XcodeGen**. This is a key architectural decision that offers several advantages:
-   **Single Source of Truth:** The YAML file is human-readable and version-controllable, preventing common issues with `.xcodeproj` merge conflicts.
-   **Consistency:** Ensures all developers generate the exact same project structure.
-   **Maintainability:** Adding new targets, files, or dependencies is a straightforward text edit rather than a complex process in the Xcode UI.

The project targets **iOS 16.0+** and is built with **Swift 5.9**.

### 3. Core Architecture & Key Patterns

#### 3.1. Multi-Target Application Container

The architecture follows a hub-and-spoke model.
-   **Hub (`Shortless` target):** The main application. It provides the user interface for configuring settings, displays statistics, and acts as the container that embeds and ships all the extensions.
-   **Spokes (Extension Targets):** Each extension is a separate binary that runs in its own process, invoked by the system to perform a specific task (e.g., blocking content in Safari, monitoring device activity).

#### 3.2. Shared Logic via Local Swift Package (`ShortlessKit`)

This is the most critical architectural pattern for code reuse and consistency. The `ShortlessKit` package is a local dependency for the main app and nearly all of its extensions. Its responsibilities include:
-   **Shared Models:** Defining data structures used across targets, such as `ScheduleRule` for Screen Time.
-   **Shared Constants:** Centralizing keys (e.g., `SettingsStore.appGroupID`, `SettingsStore.scheduleKey`) and identifiers (`DeviceActivityName.shortlessFocus`) to prevent typos and ensure consistency.
-   **Business Logic:** Encapsulating logic for reading/writing shared settings.
-   **Services:** Potentially containing network clients or other services (though this app appears to be fully on-device).

#### 3.3. Inter-Process Communication (IPC) via App Group

Since each extension runs in a separate sandbox, a mechanism is needed to share data. The project uses an **App Group** to create a shared container.
-   **Mechanism:** `UserDefaults(suiteName: SettingsStore.appGroupID)` is used to read and write simple key-value data (like settings toggles, schedules, and user selections) to a shared `plist` file within the group container.
-   **Data Flow:** The main app writes user preferences to the shared `UserDefaults`. The extensions then read these preferences at runtime to configure their behavior. For example, the `DeviceActivityMonitorExtension` reads the user's blocking schedule and app selections to know which applications to shield.

### 5. Screen Time Integration

The project leverages Apple's Screen Time APIs, which involves three distinct extensions working together:
1.  **Main App:** Uses the `FamilyControls` framework to let the user pick which apps and categories to block. It saves this selection (`FamilyActivitySelection`) into the shared `UserDefaults`.
2.  **`ShortlessActivityMonitor`:** The system wakes this extension up based on a schedule (`DeviceActivitySchedule`) set by the main app. It reads the user's selection from `UserDefaults` and uses the `ManagedSettingsStore` to apply "shields" to the selected apps and categories.
3.  **`ShortlessShieldConfig`:** When a user taps on a shielded app, the system loads this extension to provide a custom view, overriding the default system block screen.
