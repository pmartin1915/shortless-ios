# Audit Findings

An automated audit of the "shortless-ios" project has been performed.

**As of:** 2026-04-22 (UTC)

## Executive Summary

The shortless-ios project is well-structured, leveraging a shared `ShortlessKit` package and multiple extensions to deliver its features. The codebase is modern and adheres to Swift best practices. The audit identified one medium-severity logic issue related to state synchronization, two low-severity issues concerning error handling and documentation, and three informational findings for improving code quality and maintainability.

The most significant finding is a potential state mismatch in the `DeviceActivityMonitorExtension`, where changes made in the main app are not immediately reflected in the active blocking schedule, potentially leading to user confusion. Recommendations focus on improving robustness, maintainability, and ensuring user-facing documentation is synchronized with the application's features.

## Findings

### 1. State Synchronization in Device Activity Monitor

**Severity:** Medium
**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The `DeviceActivityMonitorExtension` reads its configuration (schedule, app selection, blocking mode) from `UserDefaults` only at the start or end of a `DeviceActivity` interval. If a user changes these settings in the main app while an interval is already active, the monitor will not be aware of these changes. This can lead to behavior that is inconsistent with the user's current settings.

**Example Scenario:**
1.  A blocking schedule starts at 9 AM. `intervalDidStart` runs and shields are applied based on the current settings.
2.  At 10 AM, the user opens the Shortless app and disables blocking for the current day.
3.  The shields will remain active until the interval ends (e.g., at 5 PM), even though the user has disabled the feature. This creates a confusing user experience.

**Recommendation:**
When settings related to the Screen Time schedule are modified in the main app, the app should cancel the existing `DeviceActivity` schedule and re-register it with the updated configuration. This ensures the `DeviceActivityMonitor` is always operating on the latest user-defined settings.

```swift
// In the main app, after settings change:
let center = DeviceActivityCenter()
// It's safe to call this even if no schedule is active
try await center.stopMonitoring([.shortlessFocus]) 
// ... then re-start monitoring with the new schedule and events.
```

### 2. Outdated User-Facing Documentation

**Severity:** Low
**File:** `docs/index.html`

**Description:**
The FAQ section of the support documentation contains outdated information regarding the app's capabilities. It states that native app blocking is a future feature, which contradicts the project's current state.

**Quote from `docs/index.html`:**
> "Native TikTok app blocking is coming in a future update that uses Apple's Screen Time API. In the current version, Shortless blocks TikTok in Safari only."

The presence and implementation of the `ShortlessActivityMonitor` target, which uses the Screen Time API (`DeviceActivityMonitor`), and the `ai/STATE.md` file's note that "v3.0.0 with Screen Time integration is code-complete" confirm that this feature is now implemented.

**Recommendation:**
Update the FAQ in `docs/index.html` to accurately describe the new Screen Time blocking functionality available in version 3.0.0. This will prevent user confusion and support requests related to the new feature.

### 3. Silent Failures on Data Decoding

**Severity:** Low
**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The extension decodes `ScheduleRule` and `FamilyActivitySelection` from `UserDefaults` using `try?`. If decoding fails (e.g., due to data corruption or a data model migration issue from a previous app version), the operation returns `nil`, and the function silently exits. This means app blocking will fail to activate without any indication of what went wrong, making debugging difficult.

**Code Snippet:**
```swift
// ...
guard let schedule = try? JSONDecoder().decode(ScheduleRule.self, from: scheduleData) else {
    // Fails silently if scheduleData is corrupt or in an old format
    return 
}
// ...
```

**Recommendation:**
While extensions have limited capabilities, they can use `os.log` for debugging. Add logging to capture decoding errors. This will provide crucial visibility into silent failures when troubleshooting user-reported issues.

```swift
// Recommended change
import os.log

private let logger = Logger(subsystem: "dev.pmartin1915.shortless.ActivityMonitor", category: "DataDecoding")

// ... inside intervalDidStart
if let scheduleData = defaults?.data(forKey: SettingsStore.scheduleKey) {
    do {
        let schedule = try JSONDecoder().decode(ScheduleRule.self, from: scheduleData)
        // ... continue logic
    } catch {
        logger.error("Failed to decode ScheduleRule: \(error.localizedDescription)")
        return
    }
}
```

### 4. Hardcoded String for Blocking Mode

**Severity:** Informational
**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The blocking mode is checked against a hardcoded string literal `"alwaysOn"`. This string is likely also used in the main application to set the value. Using string literals for shared keys or values is fragile; a typo in one location can lead to silent logic errors.

**Code Snippet:**
```swift
let mode = defaults?.string(forKey: SettingsStore.blockingModeKey) ?? "alwaysOn"
if mode == "alwaysOn" {
    return
}
```

**Recommendation:**
Define shared `UserDefaults` keys and critical values as static constants in a shared location, such as a struct within `ShortlessKit`. This improves type safety, enables autocompletion, and prevents typos.

```swift
// In ShortlessKit/SettingsStore.swift
public enum BlockingMode: String {
    case scheduled
    case alwaysOn
}

// In DeviceActivityMonitorExtension.swift
let modeValue = defaults?.string(forKey: SettingsStore.blockingModeKey)
let mode = BlockingMode(rawValue: modeValue ?? "") ?? .alwaysOn

if mode == .alwaysOn {
    return
}
```

### 5. Redundant `UserDefaults` Initialization

**Severity:** Informational
**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
An instance of `UserDefaults` is created at the beginning of both `intervalDidStart(for:)` and `intervalDidEnd(for:)`. This is slightly redundant.

**Recommendation:**
For improved code clarity and to adhere to the Don't Repeat Yourself (DRY) principle, initialize `UserDefaults` once as a private stored property of the `DeviceActivityMonitorExtension` class.

```swift
class DeviceActivityMonitorExtension: DeviceActivityMonitor {

    private let store = ManagedSettingsStore(named: .shortless)
    private let defaults = UserDefaults(suiteName: SettingsStore.appGroupID)

    override func intervalDidStart(for activity: DeviceActivityName) {
        // ... use self.defaults
    }

    override func intervalDidEnd(for activity: DeviceActivityName) {
        // ... use self.defaults
    }
}
```

### 6. Potentially Missing Dependency for `ShortlessShieldConfig`

**Severity:** Informational
**File:** `project.yml`

**Description:**
The `ShortlessShieldConfig` target, which provides the UI for blocked applications, does not have a dependency on the shared `ShortlessKit` package. All other app extensions in the project depend on `ShortlessKit`. While the shield configuration might be simple enough to not require shared logic currently, it is likely that it will need access to shared assets, strings, or constants in the future to maintain a consistent brand and user experience.

**Recommendation:**
Review the implementation of `ShortlessShieldConfigExtension`. If it could benefit from shared resources, or for the sake of consistency and future-proofing, consider adding `ShortlessKit` as a dependency to the `ShortlessShieldConfig` target in `project.yml`.
