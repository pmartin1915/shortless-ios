# Audit Findings

An automated audit of the "shortless-ios" project has been performed.

**Date:** 2026-04-24 (UTC)
**Project:** shortless-ios
**Auditor:** Gemini 2.5 Pro

## Executive Summary

The shortless-ios project is mature, well-structured, and leverages modern Swift and iOS practices, including XcodeGen for project management and a local Swift Package for shared code. The codebase is clean and follows platform conventions for implementing app extensions.

This audit focused on the new Screen Time integration implemented in the `ShortlessActivityMonitor` extension. The review identified one high-severity logic bug that causes the user's blocking schedule to be ignored under certain conditions. A medium-severity issue was also found where an unset configuration could lead to indefinite blocking, contrary to user expectations for a scheduled block.

Additional low-severity and informational findings include recommendations for improved error handling, code quality enhancements, and an update to user-facing documentation to reflect the new capabilities. No security vulnerabilities were identified.

## Findings

### 1. Blocking Schedule is Bypassed if Not Configured

**Severity:** <span style="color:red">High</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
In `intervalDidStart(for:)`, the logic to check the user's blocking schedule does not correctly handle the case where a schedule has not yet been saved to `UserDefaults`. If `defaults?.data(forKey: SettingsStore.scheduleKey)` returns `nil`, the `if let` block is skipped, and execution continues. The code then proceeds to fetch the `FamilyActivitySelection` and apply the shields, effectively activating blocking even on days the user has not configured for blocking.

**Impact:**
This bug breaks the core scheduling feature. If a user selects apps to block but has not configured a specific schedule, the app will block those apps 24/7 whenever a `DeviceActivity` interval is active, ignoring the user's intent for scheduled-only blocking.

**Recommendation:**
The logic should be inverted to ensure that blocking is only applied if a valid schedule exists and the current day is part of that schedule.

```swift
// In DeviceActivityMonitorExtension.swift -> intervalDidStart(for:)

// 1. Fetch schedule and selection data first
let defaults = UserDefaults(suiteName: SettingsStore.appGroupID)
guard let scheduleData = defaults?.data(forKey: SettingsStore.scheduleKey),
      let schedule = try? JSONDecoder().decode(ScheduleRule.self, from: scheduleData),
      let selectionData = defaults?.data(forKey: SettingsStore.appBlockerSelectionKey),
      let selection = try? JSONDecoder().decode(FamilyActivitySelection.self, from: selectionData) else {
    // If schedule or selection is missing, do not apply shields.
    return
}

// 2. Check if today is an active day
let today = Calendar.current.component(.weekday, from: Date())
guard schedule.activeDays.contains(today) else {
    // Not a scheduled day, do nothing.
    return
}

// 3. Apply shields
store.shield.applications = selection.applicationTokens
if !selection.categoryTokens.isEmpty {
    store.shield.applicationCategories = .specific(selection.categoryTokens)
}
```

### 2. Indefinite Blocking Occurs if Blocking Mode is Unset

**Severity:** <span style="color:orange">Medium</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
In `intervalDidEnd(for:)`, the blocking mode is retrieved from `UserDefaults`. If the key `SettingsStore.blockingModeKey` is not present, the code defaults to `"alwaysOn"`. This causes the function to `return` without removing the shields.

```swift
// Problematic default value
let mode = defaults?.string(forKey: SettingsStore.blockingModeKey) ?? "alwaysOn"

if mode == "alwaysOn" {
    return // Shields are not removed
}
```

**Impact:**
If a user is using the scheduling feature (which implies a mode other than "always on") and this setting is missing or gets cleared, the shields will never be removed when the scheduled interval ends. This results in the selected apps being blocked indefinitely until the user manually changes the mode in the main app.

**Recommendation:**
The default behavior should be to un-shield applications at the end of an interval unless the mode is explicitly `"alwaysOn"`. A safer default would be `"scheduled"` or any other string that does not match `"alwaysOn"`.

```swift
// In DeviceActivityMonitorExtension.swift -> intervalDidEnd(for:)

// A safer default, e.g., "scheduled" or an empty string.
let mode = defaults?.string(forKey: SettingsStore.blockingModeKey) ?? "scheduled"

if mode == "alwaysOn" {
    return
}

// This code will now correctly execute for scheduled mode or if the setting is missing.
store.shield.applications = nil
store.shield.applicationCategories = nil
```

### 3. Use of Magic String for Blocking Mode

**Severity:** <span style="color:blue">Low</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The blocking mode is checked using the literal string `"alwaysOn"`. This "magic string" approach is prone to typos and makes the code harder to maintain. If the string value changes, it must be updated in multiple places.

**Impact:**
This is a code quality issue that increases the risk of future bugs. A typo in either the writing or reading of this `UserDefaults` key would cause the logic to fail silently.

**Recommendation:**
Define the blocking modes as constants or an enum in the shared `ShortlessKit` package. This ensures type safety and a single source of truth.

**Example (in ShortlessKit):**
```swift
public enum BlockingMode: String {
    case alwaysOn = "alwaysOn"
    case scheduled = "scheduled"
}

// In SettingsStore.swift (or another shared constants file)
public static let blockingModeKey = "blockingMode"
```

**Usage (in DeviceActivityMonitorExtension.swift):**
```swift
let modeRawValue = defaults?.string(forKey: SettingsStore.blockingModeKey)
let mode = BlockingMode(rawValue: modeRawValue ?? "scheduled")

if mode == .alwaysOn {
    return
}
```

### 4. Silent Decoding Failures

**Severity:** <span style="color:blue">Low</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The code uses `try? JSONDecoder().decode(...)` to decode `ScheduleRule` and `FamilyActivitySelection` from `UserDefaults`. If decoding fails (e.g., due to data corruption or a data model mismatch after an update), the operation fails silently, and the function returns early. While this prevents a crash, it provides no information for debugging.

**Impact:**
If the blocking feature stops working due to a decoding error, it will be difficult to diagnose the root cause without any logs.

**Recommendation:**
For improved diagnostics, use a `do-catch` block to log any decoding errors. The `os.Logger` framework is well-suited for this purpose in app extensions.

```swift
import os.log

class DeviceActivityMonitorExtension: DeviceActivityMonitor {
    private let logger = Logger(subsystem: "dev.pmartin1915.shortless.ActivityMonitor", category: "Decoding")
    // ...

    override func intervalDidStart(for activity: DeviceActivityName) {
        // ...
        do {
            let schedule = try JSONDecoder().decode(ScheduleRule.self, from: scheduleData)
            // ... continue logic
        } catch {
            logger.error("Failed to decode ScheduleRule: \(error.localizedDescription)")
            return
        }
        // ...
    }
}
```

### 5. Outdated FAQ Regarding Native App Blocking

**Severity:** <span style="color:gray">Informational</span>

**File:** `docs/index.html`

**Description:**
The "Frequently Asked Questions" section of the support page contains the following text:
> **Can I block TikTok natively (not just in Safari)?**
> Native TikTok app blocking is coming in a future update that uses Apple's Screen Time API. In the current version, Shortless blocks TikTok in Safari only.

The `ShortlessActivityMonitor` extension is the implementation of this native app blocking feature. Since the project's state indicates that v3.0.0 with Screen Time integration is "code-complete," this documentation is now out of date.

**Impact:**
Users reading the support page for the upcoming v3.0.0 release will be misinformed about the app's capabilities.

**Recommendation:**
Update the `docs/index.html` file to reflect the new Screen Time blocking functionality before the v3.0.0 release. The question can be removed or rephrased to explain how the new feature works.
