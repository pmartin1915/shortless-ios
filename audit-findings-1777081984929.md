# Audit Findings

An automated audit of the "shortless-ios" project has been performed.

**Date:** 2026-04-25 (UTC)
**Project:** shortless-ios
**Auditor:** gemini-2.5-pro

## Executive Summary

The shortless-ios project is well-structured, leveraging XcodeGen for a clean project definition and a local Swift Package (`ShortlessKit`) for shared logic across its multiple extensions. The overall code quality in the reviewed file is good.

The audit identified one high-severity issue related to outdated user-facing documentation that misrepresents the app's current capabilities, which could lead to user confusion. Additionally, two medium-severity resilience issues were found in the Screen Time monitoring extension that could cause scheduled blocking to fail silently under certain conditions.

Recommendations focus on updating the documentation to reflect the new Screen Time features and improving the robustness of the `DeviceActivityMonitorExtension` by adding explicit error handling and clarifying assumptions in the code.

## Findings

### 1. Outdated User-Facing Documentation

| | |
| :--- | :--- |
| **Severity** | **High** |
| **File** | `docs/index.html` |
| **Category** | Content Accuracy |

**Description:**
The FAQ section in the support documentation (`docs/index.html`) incorrectly states that native app blocking is a future feature. The question "Can I block TikTok natively (not just in Safari)?" is answered with: "Native TikTok app blocking is coming in a future update that uses Apple's Screen Time API."

However, the project structure (`project.yml`), source code (`ShortlessActivityMonitorExtension.swift`), and state file (`ai/STATE.md`) all confirm that Screen Time integration for native app blocking is already implemented and code-complete for version 3.0.0. This discrepancy will confuse users who have downloaded the latest version and are looking for help with the app-blocking feature.

**Recommendation:**
Update the content in `docs/index.html` to accurately describe the current Screen Time functionality. The answer should be changed to explain that native app blocking *is* available via the Screen Time integration and briefly guide users on how to set it up within the app.

### 2. Silent Failures on Data Decoding in Activity Monitor

| | |
| :--- | :--- |
| **Severity** | **Medium** |
| **File** | `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift` |
| **Category** | Bug / Resilience |

**Description:**
In `DeviceActivityMonitorExtension.swift`, the `intervalDidStart` function decodes `ScheduleRule` and `FamilyActivitySelection` objects from `UserDefaults` using `try?`.

```swift
// Lines 18-19
if let scheduleData = defaults?.data(forKey: SettingsStore.scheduleKey),
   let schedule = try? JSONDecoder().decode(ScheduleRule.self, from: scheduleData) {
    // ...
}

// Lines 24-25
guard let data = defaults?.data(forKey: SettingsStore.appBlockerSelectionKey),
      let selection = try? JSONDecoder().decode(FamilyActivitySelection.self, from: data) else {
    return
}
```

If the JSON decoding fails for any reason (e.g., corrupted data, data model mismatch after an app update), the operation fails silently. The extension will simply `return`, and the user's selected apps will not be blocked as scheduled. The user will not be notified of this failure, making the issue difficult to diagnose.

**Recommendation:**
Replace `try?` with a `do-catch` block to handle potential decoding errors gracefully. While user-facing alerts are not possible from this extension, logging the error using `os_log` would provide crucial debugging information for the developer.

```swift
// Recommended Change
do {
    let selection = try JSONDecoder().decode(FamilyActivitySelection.self, from: data)
    // ... apply shields
} catch {
    // Use os_log or another logging mechanism to record the failure
    // for easier debugging.
    os_log("Failed to decode FamilyActivitySelection: %{public}@", log: .default, type: .error, error.localizedDescription)
}
```

### 3. Ambiguous Weekday Logic in Schedule Check

| | |
| :--- | :--- |
| **Severity** | **Medium** |
| **File** | `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift` |
| **Category** | Bug / Ambiguity |

**Description:**
The `intervalDidStart` function checks if the current day is an active day in the user's schedule with the following code:

```swift
// Lines 20-21
let today = Calendar.current.component(.weekday, from: Date())
guard schedule.activeDays.contains(today) else { return }
```

The integer value returned by `Calendar.current.component(.weekday, ...)` depends on the user's locale and calendar settings. For example, in the US, Sunday is `1`, while in France, Monday is `1`. If the `schedule.activeDays` array is populated in the main app without accounting for this, the schedule enforcement may be off by a day for some users.

**Recommendation:**
Ensure that the integer representation for weekdays is handled consistently between the main app (where the schedule is set) and this extension. A robust approach is to use a specific calendar, such as `Calendar(identifier: .iso8601)`, which consistently uses 1 for Monday through 7 for Sunday, regardless of locale. Alternatively, add a comment to the code clarifying the expected integer-to-day mapping (e.g., "1 = Sunday, ... 7 = Saturday") and ensure the main app adheres to it.

### 4. Inconsistent Target Dependency

| | |
| :--- | :--- |
| **Severity** | **Low** |
| **File** | `project.yml` |
| **Category** | Code Quality / Configuration |

**Description:**
In `project.yml`, the `ShortlessShieldConfig` target does not have a dependency on the `ShortlessKit` package, whereas every other app extension target does. While this extension might be simple enough to not require shared code, it breaks the established architectural pattern. It is likely that it will need access to shared assets, constants, or helper functions from `ShortlessKit` in the future.

**Recommendation:**
Add a dependency on `ShortlessKit` to the `ShortlessShieldConfig` target for consistency and to facilitate future use of shared code.

```yaml
# In project.yml
ShortlessShieldConfig:
  type: app-extension
  platform: iOS
  sources: ShortlessShieldConfig
  dependencies: # Add this section
    - package: ShortlessKit
  settings:
    # ...
```

### 5. Unit Tests Depend on Application Target

| | |
| :--- | :--- |
| **Severity** | **Informational** |
| **File** | `project.yml` |
| **Category** | Code Quality / Best Practice |

**Description:**
The `ShortlessTests` target lists a dependency on the main application target `Shortless`.

```yaml
# In project.yml
ShortlessTests:
  # ...
  dependencies:
    - target: Shortless
    - package: ShortlessKit
```

While this is a valid configuration, it is generally better for unit tests to depend on the specific library or framework containing the business logic (`ShortlessKit` in this case) rather than the full application. This can reduce build times for the test suite and encourages a clear separation between logic and UI. UI-specific tests can be handled in a separate UI Test target.

**Recommendation:**
Consider refactoring the unit tests to only depend on `ShortlessKit`. If tests require access to code within the `ShortlessApp/` source directory, that code may be a candidate for relocation into `ShortlessKit` to be more easily testable and reusable.
