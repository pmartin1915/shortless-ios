# Audit Findings

An automated audit of the "shortless-ios" project has been performed.

**Date:** 2026-04-25 (UTC)
**Project:** shortless-ios
**Auditor:** Gemini 2.5 Pro (Automated)

## Executive Summary

The audit of the "shortless-ios" project reveals a well-structured application with multiple extensions and a shared core logic module (`ShortlessKit`). The codebase is generally clean and follows modern Swift practices.

However, a high-severity functional bug was identified in the `DeviceActivityMonitorExtension` that causes the "Always On" blocking mode to fail on days not included in the user's schedule. Additionally, the user-facing documentation in `docs/index.html` is outdated and misrepresents the app's current capabilities regarding Screen Time integration, which could lead to user confusion and support requests.

Other findings include opportunities to improve code robustness by adding logging for silent failures and enhancing project configuration for better maintainability.

Overall, the project is in a good state, but the identified high-severity bug should be addressed before the next release to ensure core functionality works as intended.

## Findings

### 1. "Always On" Blocking Mode is Not Respected on Non-Scheduled Days

**Severity:** <span style="color:red">**High**</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
In `DeviceActivityMonitorExtension.swift`, the `intervalDidStart` function checks if the current day is an active day in the user's schedule *before* checking the blocking mode.

```swift
// ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift:16
override func intervalDidStart(for activity: DeviceActivityName) {
    // ...
    // Check if today is an active day in the schedule
    if let scheduleData = defaults?.data(forKey: SettingsStore.scheduleKey),
       let schedule = try? JSONDecoder().decode(ScheduleRule.self, from: scheduleData) {
        let today = Calendar.current.component(.weekday, from: Date())
        guard schedule.activeDays.contains(today) else { return } // <--- BUG HERE
    }

    guard let data = defaults?.data(forKey: SettingsStore.appBlockerSelectionKey),
          // ...
}
```

If the user has selected the "Always On" blocking mode, their intent is for blocking to be active every day, regardless of the schedule. However, due to this check, if an interval starts on a day that is not in `schedule.activeDays`, the function returns early and shields are never applied.

**Impact:**
The "Always On" feature, a core part of the app's functionality, is broken. Users who select this mode will find that blocking is unexpectedly disabled on weekends or any other day they have not explicitly included in their schedule, defeating the purpose of the feature.

**Recommendation:**
Refactor `intervalDidStart` to check the `blockingMode` first. If the mode is `"alwaysOn"`, the schedule check should be bypassed entirely.

```swift
// Suggested Fix
override func intervalDidStart(for activity: DeviceActivityName) {
    guard activity == .shortlessFocus else { return }

    let defaults = UserDefaults(suiteName: SettingsStore.appGroupID)
    let mode = defaults?.string(forKey: SettingsStore.blockingModeKey) ?? "alwaysOn"

    // If not in "alwaysOn" mode, check the schedule
    if mode != "alwaysOn" {
        if let scheduleData = defaults?.data(forKey: SettingsStore.scheduleKey),
           let schedule = try? JSONDecoder().decode(ScheduleRule.self, from: scheduleData) {
            let today = Calendar.current.component(.weekday, from: Date())
            guard schedule.activeDays.contains(today) else { return }
        }
    }

    // Apply shields (common logic for both modes)
    guard let data = defaults?.data(forKey: SettingsStore.appBlockerSelectionKey),
          let selection = try? JSONDecoder().decode(FamilyActivitySelection.self, from: data) else {
        return
    }

    store.shield.applications = selection.applicationTokens
    if !selection.categoryTokens.isEmpty {
        store.shield.applicationCategories = .specific(selection.categoryTokens)
    }
}
```

### 2. Outdated User-Facing Documentation

**Severity:** <span style="color:orange">**Medium**</span>

**File:** `docs/index.html`

**Description:**
The FAQ section in the support documentation contains outdated information about the app's capabilities.

```html
<!-- docs/index.html:202 -->
<h3>Can I block TikTok natively (not just in Safari)?</h3>
<p>Native TikTok app blocking is coming in a future update that uses Apple's Screen Time API. In the current version, Shortless blocks TikTok in Safari only.</p>
```

This statement directly contradicts the project's current state. The presence of `ShortlessActivityMonitor` and `ShortlessShieldConfig` targets, along with the logic in `DeviceActivityMonitorExtension.swift`, confirms that native app blocking via the Screen Time API is already implemented and code-complete for version 3.0.0 (as per `ai/STATE.md`).

**Impact:**
This discrepancy will confuse users, misrepresent the product's value, and likely lead to unnecessary support requests and negative App Store reviews. Users may not realize the native blocking feature exists or may be confused when trying to set it up.

**Recommendation:**
Update the content of `docs/index.html` to accurately reflect the app's current features. The FAQ should be rewritten to explain *how* to use the new Screen Time integration for native app blocking, rather than stating it is a future feature.

### 3. Silent Failures on Data Decoding

**Severity:** <span style="color:blue">**Low**</span>

**File:** `ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift`

**Description:**
The monitor extension uses `try?` to decode `ScheduleRule` and `FamilyActivitySelection` from `UserDefaults`. If decoding fails for any reason (e.g., data corruption, model mismatch after an update), the function returns silently.

```swift
// ShortlessActivityMonitor/DeviceActivityMonitorExtension.swift:24
guard let selection = try? JSONDecoder().decode(FamilyActivitySelection.self, from: data) else {
    return
}
```

While this prevents a crash, it provides no diagnostic information. When a user reports that blocking has mysteriously stopped working, it will be very difficult for the developer to debug the issue without any logs.

**Impact:**
Increased difficulty in diagnosing and fixing user-reported issues where blocking fails to activate. The app fails silently, leaving both the user and developer without information.

**Recommendation:**
Incorporate logging to record decoding failures. Using a framework like `OSLog` is lightweight and standard practice for iOS extensions.

```swift
// Suggested Improvement
import os.log

// At class level
private let logger = Logger(subsystem: "dev.pmartin1915.shortless.ActivityMonitor", category: "Decoding")

// In intervalDidStart
guard let selection = try? JSONDecoder().decode(FamilyActivitySelection.self, from: data) else {
    logger.error("Failed to decode FamilyActivitySelection from UserDefaults.")
    return
}
```

### 4. Missing Dependency for Shield Configuration Target

**Severity:** <span style="color:gray">**Informational**</span>

**File:** `project.yml`

**Description:**
In the `project.yml` file, the `ShortlessShieldConfig` target does not declare a dependency on the shared `ShortlessKit` package, while all other app extensions do.

```yaml
# project.yml:195
ShortlessShieldConfig:
  type: app-extension
  platform: iOS
  sources: ShortlessShieldConfig
  # No dependencies listed
```

The Shield Configuration extension is responsible for the UI presented to the user on a blocked app. It is highly likely that this UI would need to access shared constants, strings, colors, or other assets from `ShortlessKit` to maintain a consistent user experience with the main app.

**Impact:**
This may lead to duplicated code or assets, or an inconsistent UI. It deviates from the established architectural pattern used by all other extensions in the project.

**Recommendation:**
Add `ShortlessKit` as a dependency for the `ShortlessShieldConfig` target in `project.yml` to promote code reuse and consistency.

```yaml
# Suggested Fix
ShortlessShieldConfig:
  type: app-extension
  platform: iOS
  sources: ShortlessShieldConfig
  dependencies:
    - package: ShortlessKit
  # ... rest of the definition
```
